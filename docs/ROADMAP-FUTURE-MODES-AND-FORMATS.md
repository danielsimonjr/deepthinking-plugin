# Roadmap: Reasoning Modes and Output Formats — Future Work

> Generated 2026-04-13 by RLM forward-looking inventory pass over 101 reference docs (plugin + deprecated MCP source corpus). 350+ raw findings synthesized via Sonnet.

## Methodology

This roadmap was generated by scanning 101 reference files from the deepthinking-plugin codebase and deprecated MCP source corpus. A parallel Haiku pass extracted 350+ mentions of reasoning methods and output formats not present in the canonical sets. These raw findings were then filtered against the current 34 reasoning modes and 9 format grammars to identify genuine gaps versus already-covered functionality.

Findings were categorized into three confidence levels: Strong candidates have multiple citations or fill clear gaps in the existing taxonomy; Plausible candidates have single citations but represent defensible extensions; Rejected candidates either duplicate existing functionality or fall outside the plugin's scope. Each entry includes effort estimates (Trivial/Small/Medium/Large) and redundancy risk assessments to guide implementation prioritization.

The analysis focused on methods with established academic foundations or clear software engineering applications, filtering out overly narrow techniques or those requiring significant runtime infrastructure beyond the plugin's current architecture.

### Methodology Notes — Post-synthesis verification corrections

The Sonnet synthesis pass produced strong but imperfect output. Manual verification against the actual codebase caught four cases where the synthesis misclassified items:

1. **`toulmin` was originally proposed as Tier 1** — but `skills/think-academic/SKILL.md` defines the canonical `argumentation` mode AS the Toulmin model (5 references, including the worked example). Moved to Tier 3 (already canonical).
2. **`socratic` was originally proposed as Tier 1** — but the same skill file lists Socratic questioning as a component of canonical `critique`. Moved to Tier 3 (already canonical).
3. **`mermaid` and `dot` were originally proposed as Tier 1 formats "to be promoted from render-diagram.py"** — but they ARE already first-class formats. They live at `reference/visual-grammar/<mode>.md` (per-mode templates, one file per mode) rather than `reference/visual-grammar/formats/<format>.md` (mode-agnostic surface syntax) because each of the 34 modes needs its own Mermaid+DOT template tailored to its semantic structure. The README correctly lists 11 total formats (the 9 in `formats/` plus mermaid+dot from per-mode files). Moved to Tier 3 with an architectural explanation.
4. **`structuralcausalmodel` was originally Tier 1** — but Pearl's SCMs are largely a formalization of what `causal` + `counterfactual` already cover. Demoted to Tier 2 (later rejected to Tier 3 in round 2 — see below).

**Lesson for future RLM-driven analyses:** the synthesis pass is high-leverage but the model has no direct access to the running codebase — so claims like "this is missing" must be verified against actual file contents before being acted on. Three of the four corrections came from grep-checking individual SKILL.md files; the fourth came from understanding the per-mode-vs-shared-format split documented in CLAUDE.md. **Always verify before recommending.**

### Methodology Notes — Round 2: Review team verdict

After the post-synthesis corrections above, the document was reviewed by a 5-specialist team (Opus architect + Opus adversary + Sonnet technical + Sonnet domain expert + Sonnet editorial) running in parallel via the same RLM `llm_query` helper. The team produced 99 findings; concordance analysis surfaced 11 items mentioned by 2+ reviewers.

**The unified verdict was that the round-1 corrections did not go far enough.** The same verification standard that caught Toulmin and Socratic ("is this actually distinct from existing modes?") was not consistently applied to the surviving Tier 1/2 mode candidates. Specifically:

- **`dialectical` failed the same test** — 4 reviewers (adversary + architect + domain + technical) flagged it as redundant with the existing `synthesis` + `argumentation` + `critique` trio. Adversary verbatim: "dialectical is the same pattern" as toulmin/socratic — a framework already covered inside parent modes. Demoted from Tier 1 → Tier 3 in round 2.
- **`decisionanalysis` is borderline** — adversary said "redundant with gametheory + bayesian + optimization", architect said "genuinely fills a gap". The schema sketch did look like a union of three other modes' fields. Demoted from Tier 1 → Tier 2 with an explicit risk note rather than fully rejected, pending a concrete schema design that proves distinctness.
- **`structuralcausalmodel`** — both Opus reviewers (architect AND adversary) explicitly recommended full rejection. The document's own analysis admitted it would be "better as a rigor mode extension to causal rather than a new mode." Moved Tier 2 → Tier 3 in round 2.
- **The entire computational-Tier 2 cluster** (`dynamicalsystems`, `controltheory`, `networkanalysis`, `reliabilityengineering`, `probabilistic`) was flagged for rejection on a unified principle: **the plugin teaches reasoning patterns, not computation**. These methods all require auto-computation (eigenvalues, PID tuning, centrality calculations, p-values, numerical integration) that the plugin cannot perform — they belonged to the deprecated MCP's runtime, not to the plugin's prose-only reasoning model. Moved to Tier 3 in round 2 with the new rejection rationale **"requires computation, not reasoning"**.
- **`reliabilityengineering`** is independently rejected because it is already covered by canonical `engineering` (which explicitly includes failure analysis — same reason FMEA was already in Tier 3).
- **Tier 1 modes is now empty.** This is the honest answer the round-1 corrections backed away from. The plugin's 34 modes are essentially feature-complete for prose-based structured reasoning. Future mode additions should clear a higher bar: a concrete schema sketch, a worked example that doesn't reduce to existing modes, and at least one external user request.

**Strategic gaps the team flagged that the original roadmap missed:**

| Gap | Severity | Source |
|---|---|---|
| No mention of mode-index.md maintenance burden — adding a mode requires updating an auto-recommend decision tree manually | Medium | architect |
| Smoke test scalability — already 30–60 min for 34 modes; doesn't scale to 40+ | High | architect |

**Effort estimates were systematically optimistic.** Every new mode is at least Medium because the 10-place mode invariant requires touching schema + sample + visual-grammar + output-formats + skill + router + mode-index + taxonomy + smoke prompt + MODE_DISPLAY_NAMES dict — even before content authoring. All mode effort estimates in this document were updated accordingly in round 2.

**Tone of the verdict:** Both Opus reviewers reached the same conclusion via different reasoning paths. The architect's summary called the document "conservative and principled" but flagged that "34 modes may already be at or near the sustainable ceiling." The adversary's verdict was harsher: "the plugin is closer to done than this roadmap admits" and "after applying the same verification standard the document used for toulmin/socratic, the credible v0.5.0 scope shrinks to: maybe latex-math (for rendering, not as a JSON-escape fix), maybe csv (with realistic effort). No new modes are justified until user demand materializes."

This is the honest read, and the round-2 edits reflect it. **The roadmap below is materially smaller than its first draft because the plugin is materially closer to feature-complete than the first draft assumed.**

### Methodology Notes — Round 3: HonestClaude verification pass

After the round-2 review-team edits, the document was put through the HonestClaude SKILL pass — a discipline focused on factual grounding, citation verification, and hallucination prevention (per `~/.claude/skills/HonestClaude/SKILL.md`, based on Anthropic's official "reduce hallucinations" guidance).

The pass produced **36 verifiable claims** across four rounds, checked programmatically against the actual codebase: citation existence, schema field presence, mode-list consistency, numeric counts, and method-definition factual accuracy. Results:

| Verdict | Count | Meaning |
|---|---|---|
| PASS | 33 | Claim verified by reading the actual file or running an equivalent check |
| WARN | 2 | Soft signal; investigated and resolved (see below) |
| FIX (later overturned) | 1 | False-positive in the verification check itself; the underlying claim was correct |
| FAIL | 0 | No unresolvable contradictions |

Plus a round-2 deeper dive of 45 source-citation paths from the roadmap, of which **41 of 41** verified after filtering 4 regex false positives (the words "All" and "plus" picked up from prose like "All 34 files at..., plus...").

**The single FIX item turned out to be a false positive in my verification script, not a problem with the roadmap.** The script checked for Toulmin field names (`claim`, `grounds`, `warrant`, `backing`, `qualifier`, `rebuttal`) as direct properties in the `argumentation.json` schema, but the schema correctly uses plural forms (`warrants`, `backings`, `qualifiers`, `rebuttals`) because real arguments can have multiple instances of each. A corrected check confirmed all 6 Toulmin fields are present in the schema as plurals. The roadmap's Toulmin rejection rationale stands.

**Two real corrections came out of the HonestClaude pass:**

1. **The CSV "8-12 of 34 modes" estimate was too low.** A heuristic check (counting modes whose schemas contain at least one array-of-objects field with ≥3 properties) found **27 of 34** modes have tabular content. The CSV entry was updated to reflect the empirical count and to clarify that the design pass should prioritize modes whose tabular content is *primary* (e.g., `gametheory.payoffMatrix`) rather than incidental.

2. **The structuralcausalmodel rejection got stronger.** Grep of `skills/think-causal/SKILL.md` confirmed the canonical causal mode **already teaches `do-operator` reasoning** — the term appears in the skill prose, though Pearl is not named. This is a stronger rejection than round 2 had: the plugin already has implicit do-calculus support inside the canonical causal mode. Pearl's formalism is not missing, it's unattributed. The right fix (if any) is a one-paragraph addition to the causal SKILL.md citing Pearl, not a new mode.

## Current canonical sets

**Logical & Formal:**
- abductive — inference to best explanation
- deductive — logical derivation from premises
- inductive — pattern generalization from examples
- formallogic — symbolic logic and proof construction

**Mathematical & Computational:**
- algorithmic — step-by-step problem solving
- computability — decidability and complexity analysis
- mathematics — mathematical proof and calculation
- recursive — self-referential problem decomposition

**Probabilistic & Uncertain:**
- bayesian — belief updating with evidence
- evidential — Dempster-Shafer belief functions
- stochastic — random process modeling

**Causal & Temporal:**
- causal — cause-effect relationship analysis
- counterfactual — what-if scenario reasoning
- temporal — time-based relationship reasoning
- historical — past event analysis and precedent

**Systems & Strategic:**
- systemsthinking — holistic system behavior analysis
- gametheory — strategic interaction modeling
- optimization — constraint satisfaction and improvement
- constraint — constraint satisfaction problems

**Analytical & Critical:**
- analysis — systematic decomposition
- critique — weakness identification and evaluation
- argumentation — structured argument construction
- synthesis — integration of multiple sources

**Scientific & Engineering:**
- scientificmethod — hypothesis testing methodology
- engineering — design trade-offs and failure analysis
- physics — physical law application
- firstprinciples — fundamental assumption reasoning

**Information & Communication:**
- shannon — information theory and encoding
- cryptanalytic — code breaking and security analysis

**Cognitive & Meta:**
- analogical — similarity-based reasoning
- metareasoning — reasoning about reasoning
- modal — possibility and necessity logic
- hybrid — multi-method combination
- sequential — ordered step reasoning

### 9 mode-agnostic format grammars (v0.4.1)

These live at `reference/visual-grammar/formats/<format>.md` and define surface syntax shared across all 34 modes:

- ascii — plain text diagrams and tables
- dashboard — interactive HTML with embedded visualizations
- graphml — XML graph interchange format
- html — structured web document format
- json — structured data interchange
- markdown — lightweight markup with embedded diagrams
- modelica — differential equation modeling language
- tikz — LaTeX vector graphics (TikZ package)
- uml — unified modeling language diagrams

### 2 per-mode visual grammars (v0.4.1) — also first-class

These live at `reference/visual-grammar/<mode>.md` (one file per mode) because each mode needs its own Mermaid + DOT template tailored to its semantic structure. They are equally first-class — `render-diagram.py` wraps `mmdc` and `dot` to render SVG/PNG from these:

- mermaid — diagram-as-code for flowcharts, sequence diagrams, state machines (rendered via `mmdc`)
- dot — Graphviz directed/undirected graphs (rendered via `dot`)

**Total effective format coverage: 11** (the README's "All 11 export formats" table reflects this). Plus `svg` and `png` as derived rendering targets from mermaid+dot. Future format candidates below should be evaluated against this 11-format baseline, not the 9-file `formats/` directory.

## Roadmap: Reasoning Modes

### Tier 1 — Strong candidates (merit serious consideration)

**(intentionally empty)**

After two rounds of verification (the post-synthesis pass and the 5-specialist review team), no proposed reasoning mode survived the bar that Tier 1 implies — namely: "high-confidence, well-justified, distinct from every canonical mode, with a concrete schema sketch that cannot be expressed as a special case of an existing mode."

The original draft listed `toulmin`, `socratic`, `dialectical`, and `decisionanalysis` here. The first two were rejected in round 1 because they are already inside the canonical `argumentation` and `critique` modes. The last two failed the same test in round 2: `dialectical` is the thesis-antithesis-synthesis structure already covered by combining canonical `synthesis` (multi-source integration) + `argumentation` (Toulmin) + `critique` (peer review), and `decisionanalysis` produced a schema sketch that read as a union of `gametheory` + `bayesian` + `optimization` field sets.

**This is a real finding, not a writing failure.** The plugin's 34 modes are essentially feature-complete for prose-based structured reasoning. Future mode additions should clear a higher bar:

1. A concrete JSON Schema sketch with field types and required/optional designations (not just a bag of nouns)
2. A worked example that does NOT reduce to existing modes when you remove the new mode's name
3. At least one external user request, or a documented gap in `test/smoke/captured/` outputs where a real Claude response failed to fit any existing schema
4. An explicit acknowledgment of the 10-place mode invariant cost (every new mode is at least Medium effort because of the synchronized-artifacts requirement)

If a candidate cannot meet all four bars, it belongs in Tier 2 (watching) or Tier 3 (rejected with rationale).

### Tier 2 — Watch list (re-evaluate when concrete demand or design surfaces)

**decisionanalysis** — Decision Analysis (held; needs concrete distinctness)
- **What it is** — A formal methodology for individual decision-making under uncertainty using decision trees, utility functions, expected-value calculations, sensitivity analysis, and value-of-information analysis. Foundational frameworks include Howard's decision analysis cycle and Raiffa's decision trees.
- **Why it might earn inclusion** — Plausibly fills a gap not covered by `gametheory` (multi-agent strategic interaction), `bayesian` (belief updating only), or `optimization` (constraint satisfaction with fixed objective). Decision analysis combines probabilistic belief, multi-attribute utility, and explicit alternative enumeration into a single structured artifact — none of the three canonical modes does this end-to-end.
- **Why it is held in Tier 2 not promoted** — The round-2 adversary review pointed out that the schema sketch (`alternatives, criteria, probabilities, utilities, expected_values, sensitivity_analysis, value_of_information`) reads as a union of fields from those three other modes. Until someone produces a worked example where the integrated structure adds value beyond what the three modes can each separately produce, the case for a new mode is weak. The right next step is a design spike: take a real decision (e.g., "should we migrate to PostgreSQL?") and produce both (a) three separate `bayesian`/`gametheory`/`optimization` thoughts and (b) one decisionanalysis thought, then judge whether the integrated artifact is materially better.
- **Closest existing modes** — bayesian (probabilities), optimization (alternatives + criteria), gametheory (utilities, but only multi-agent)
- **Source citations** — plugin/skills/think/SKILL.md, plugin/skills/think-probabilistic/SKILL.md, plugin/README.md (all generic infrastructure files; no source explicitly requests this mode)
- **Schema sketch (would need refinement)** — `decisionContext` (objective, decisionMaker, timeHorizon), `alternatives[]` (id, description), `uncertainNodes[]` (variable, distribution, parents), `utilityFunction` (attributes, weights, normalization), `decisionTree` (nodes, branches, terminal payoffs), `expectedValues[]` (per alternative), `sensitivityAnalysis` (which inputs swing the decision), `valueOfInformation` (per uncertain node)
- **Effort** — Medium-Large (10-place invariant + non-trivial math instruction in the SKILL.md prose; would require teaching Claude to compute expected values reliably, which is a real schema-correctness concern)
- **Risk of redundancy** — Medium (the team was split — 1 architect for, 1 adversary against)
- **Priority** — Hold for v0.6.0 or later; do not commit until a design spike demonstrates non-redundant utility

### Tier 3 — Considered and rejected

**dialectical** — Dialectical Reasoning (rejected in round 2 — was Tier 1)
- **Why rejected** — Failed the same "is this distinct from existing modes?" test that caught `toulmin` and `socratic` in round 1. Thesis-antithesis-synthesis structure is already covered by combining canonical `synthesis` (multi-source integration with convergence/divergence tracking) + `argumentation` (Toulmin claim/warrant/rebuttal) + `critique` (peer-review evaluation). The round-2 review's domain expert also flagged that "dialectical reasoning" is not a single well-defined method — it's a family of approaches (Hegelian, Platonic, Marxist) without a unifying schema. Four out of five reviewers (adversary, architect, domain, technical) recommended demotion or rejection. The schema sketch (`thesis, antithesis, synthesis, tension_points, resolution_mechanism, synthesis_quality`) was vague and `synthesis_quality` is subjective, making the validation rule unverifiable.
- **Source citations** — mcp/docs/superpowers/knowledge-packs/argumentation.md, mcp/docs/modes/ARGUMENTATION.md (the citations pointed to the existing argumentation knowledge pack, not to a separate dialectical method)

**structuralcausalmodel** — Structural Causal Models / Pearl do-calculus (rejected in round 2 — was Tier 2)
- **Why rejected** — Both Opus reviewers (architect AND adversary) explicitly recommended full rejection. The document's own analysis admitted SCMs would be "better as a rigor mode extension to causal rather than a new mode." Adding it would create a three-mode causal cluster (`causal` + `counterfactual` + `structuralcausalmodel`) with significant overlap. The architect's recommendation: if Pearl's do-calculus formalism is genuinely needed, **extend the existing `causal` schema** with optional `structuralEquations` and `interventions` fields rather than fork a new mode. The mention of "Pearl's three-rung causal hierarchy" was also flagged by the domain expert as imprecise — Pearl's "ladder of causation" has three levels (association, intervention, counterfactual), which is already exactly what the canonical `causal` + `counterfactual` modes cover between them.
- **HonestClaude verification finding** — A grep of `skills/think-causal/SKILL.md` confirms the canonical causal mode **already teaches `do-operator` reasoning** (the term appears in the skill, even though Pearl is not named directly). This makes the rejection rationale stronger than the round-2 review realized: the plugin already has implicit do-calculus support inside the canonical causal mode. Pearl's formalism is not *missing*; it's *unattributed*. If the real concern is making the Pearl lineage explicit, the right fix is a one-paragraph addition to the causal SKILL.md citing Pearl, not a new mode.
- **Source citations** — plugin/skills/think-causal/SKILL.md (the only citation, which describes the existing causal mode and contains `do-operator` reasoning per HonestClaude verification)

**dynamicalsystems**, **controltheory**, **networkanalysis**, **probabilistic** — Computational methods (rejected in round 2 — were all Tier 2)
- **Why rejected (unified rationale)** — All four require computation that the plugin cannot perform. Dynamical systems analysis needs eigenvalues and numerical integration. Control theory needs PID tuning and Bode-plot stability margins. Network analysis needs centrality calculations and community-detection algorithms. General probabilistic reasoning needs frequentist hypothesis tests, p-values, and maximum-likelihood fits. **The plugin teaches reasoning patterns expressed in JSON; it does not run computation.** This is the dividing line between the plugin and the deprecated MCP — the MCP had TypeScript handlers that auto-computed Bayes posteriors, Nash equilibria, and Allen interval relations; the plugin teaches Claude to produce these in prose. Methods whose value lies in *computation* rather than *structured reasoning about what to compute* are out of scope per the "no Node runtime" principle in CLAUDE.md.
- **Architect note** — `networkanalysis` may be revisited in the future if `graphml` consumers (Gephi, NetworkX users) start requesting structured-graph reasoning artifacts. The other three are unlikely to be reconsidered.
- **Source citations** — plugin/reference/visual-grammar/formats/modelica.md (3 of the 4); plugin/skills/think-scientific/SKILL.md, plugin/skills/think-engineering/SKILL.md, plugin/reference/visual-grammar/formats/graphml.md, plugin/skills/think-core/SKILL.md (all generic infrastructure files; none of these is a request for a new mode)

**reliabilityengineering** — Reliability Engineering (rejected in round 2 — was Tier 2)

This is the same rejection rationale the document already applies to `fmea` (Failure Mode and Effects Analysis) below. ## Roadmap: Output Formats

### Tier 1 — Strong candidates (merit serious consideration)

**latex-math** — LaTeX Math Notation (distinct from canonical TikZ)
- **What it is** — General LaTeX mathematical typesetting (equations, proofs, derivations, matrices) using the AMS-math packages — distinct from the canonical `tikz` format which is for vector diagrams using the TikZ package. The two are complementary: one renders math, the other renders graphs.
- **Consumers** — Academic papers, Overleaf, MathJax/KaTeX in web pages, local pdflatex installations. Also useful as embedded blocks inside markdown or HTML output (where the existing canonical `markdown` and `html` formats already pass through math content but don't structure it).
- **Why it earns inclusion** — Mathematics, physics, formallogic, firstprinciples, and bayesian modes all naturally produce equations as part of their structured output. A dedicated `latex-math` format grammar would standardize how math expressions are *extracted* from a thought and rendered as a clean math-only artifact (e.g., a `.tex` file that could be `\input`'d into a paper).
- **Round-2 correction** — The original draft claimed this format would "address the long-term fix flagged in `project_schema_lessons.md` for the LaTeX-in-JSON-strings issue." The adversary reviewer correctly pointed out that this is wrong: the `repair_lone_backslashes` workaround exists because Claude emits LaTeX strings inside JSON without doubling backslashes for JSON's escape rules. That is a **SKILL.md prose bug** (the math/physics/formallogic skills should explicitly instruct "escape all backslashes in JSON strings as `\\\\`") and a **schema authoring choice** (math-content fields should be plain strings with documented escape requirements). It is NOT a missing-output-format problem. Adding `latex-math` does not eliminate `repair_lone_backslashes` — those are independent fixes. The honest motivation for `latex-math` is **rendering convenience**, not **JSON-escape repair**.
- **Source citations** — mcp/docs/superpowers/knowledge-packs/firstprinciples.md, mcp/docs/superpowers/knowledge-packs/formallogic.md, mcp/docs/modes/GAMETHEORY.md
- **Surface syntax sketch** — `\begin{align*} P(H|E) &= \frac{P(E|H) \cdot P(H)}{P(E)} \\ &= \frac{0.9 \cdot 0.3}{0.41} \approx 0.659 \end{align*}` (note: the `align*` environment requires `amsmath`, so the grammar file must document this preamble dependency)
- **Renderer dependency** — None for source output. For visual rendering the user needs either `pdflatex` (with `amsmath` and `amssymb` packages) or a JavaScript-based KaTeX/MathJax renderer in a browser. No new binary becomes a hard plugin dependency, but the grammar file should document the `amsmath` requirement so users don't get cryptic LaTeX errors.
- **Effort** — Small to Medium. The grammar file itself is small (~1 file). The work is in defining per-mode math-extraction rules: which JSON fields in each of the 7 math-heavy mode schemas (mathematics, physics, computability, formallogic, firstprinciples, bayesian, evidential) should be extracted as math expressions vs prose. This is a spec-design pass that requires reading 7 schemas and 7 SKILL.md files to identify the math-bearing fields. Estimated 2-4 hours of design work plus 1-2 hours of grammar authoring.
- **Priority** — Medium-High (rendering convenience for math-heavy modes is a real user benefit; the only Tier 1 format that survived two review rounds intact)

**csv** — Comma-Separated Values
- **What it is** — Universal tabular data format for structured numerical and categorical output. Useful when a thought contains tabular subsections (decision matrices, payoff tables, evidence weights, alternative comparisons) that need to be exported for downstream analysis tools.
- **Consumers** — Excel, R, Python pandas, database import tools, statistical software, jq pipelines, awk scripts, anything that consumes RFC 4180 CSV
- **Why it earns inclusion** — Several canonical modes naturally produce tabular data: `gametheory.payoffMatrix.entries`, `bayesian.evidence` rows, `optimization.alternatives`, `engineering.tradeStudy.alternatives`, `evidential.belief` distributions, `decisionanalysis.expectedValues` (if that mode ships). A CSV format grammar would let users pipe these directly into spreadsheet/dataframe tools without writing extraction code.
- **Round-2 correction** — The original draft estimated this as Trivial. The technical reviewer correctly pointed out that a CSV format grammar isn't just surface syntax — **it needs flattening rules for each of the 34 mode schemas** to decide which JSON fields become CSV rows, which become columns, and how nested structures are flattened (column-per-key? JSON-string cell? skipped?). This is a spec-design pass, not a one-line definition. The honest estimate is **Small** (not Trivial), with maybe **half a day** of decisions about which modes actually have CSV-meaningful tabular content. (Empirical heuristic from a HonestClaude pass: **27 of 34 modes** have at least one array-of-objects field with ≥3 properties — most modes are tabular-friendly. The grammar design pass would prioritize the modes whose tabular content is the *primary* artifact rather than incidental, e.g., `gametheory` payoff matrices and `bayesian` evidence rows over `analogical` mapping arrays.)
- **Source citations** — plugin/reference/visual-grammar/formats/modelica.md (the only direct citation; the rationale above is from analyzing the 34 schemas)
- **Surface syntax sketch** — For a `bayesian` thought with hypothesis + 3 evidence items: ```evidence_id,description,prior,likelihood_h,likelihood_not_h,posterior\nE1,observed bug pattern X,0.30,0.90,0.20,0.66\nE2,...```. For a `gametheory` thought, it would be the payoff matrix flattened as `player1_strategy,player2_strategy,player1_payoff,player2_payoff`.
- **Renderer dependency** — None (text output, no external binary)
- **Effort** — Small (was: Trivial)
- **Priority** — Medium

## Sequencing recommendation

After two rounds of review (post-synthesis verification + 5-specialist team review), the credible roadmap is dramatically smaller than the initial Sonnet draft suggested. **No new reasoning modes are recommended for v0.5.0.** The plugin's 34 modes are essentially feature-complete; new modes should require external user demand rather than internal speculation. Only output formats survived the second review pass.

### v0.5.0 — Format expansion only (no new modes)

- **`latex-math`** format grammar (Small-Medium effort, ~half a day of design work + 1-2 hours of grammar authoring). Provides clean math rendering for the 7 math-heavy modes.
- **`csv`** format grammar (Small effort, was Trivial — corrected to account for per-mode flattening rules). Provides tabular export for the ~8-12 modes that produce naturally tabular data.

That's it. Two format grammars, both small, both addressing real consumer needs (academic/publication workflows for `latex-math`; data-analysis-tool integration for `csv`). Total effort: 1-2 days of focused work for one engineer, including testing, sample fixtures, and `test/test_format_grammars.py` updates.

### v0.5.x patches — quality and tooling (not new functionality)

These are not on the original roadmap but emerged from the round-2 strategic-gaps review. They have higher leverage than any new mode:

- **`test_version_consistency.py`** — cross-file version coherence check (was flagged as a v0.5.0 deferred item in `memory/project_key_discoveries.md`)
- **`MODE_DISPLAY_NAMES` automated sync check** — extend `test_artifact_consistency.py` to verify the dict in `scripts/render-html-dashboard.py` matches the canonical mode set (the 10th sync location flagged in CLAUDE.md)
- **Smoke test parallelization** — currently sequential at 30-60 min; with 4 parallel `claude -p` workers this drops to ~10-15 min. Architect flagged this as **High severity** because the current test cost makes any future mode addition expensive to verify.
- **`examples/personal-command-alias/think-render.md`** — ship the missing alias parallel to the existing `think.md` (1-line task, mentioned in `memory/project_invocation_forms.md`)

### v0.6.0 — Speculative; do not commit until evidence exists

The team explicitly flagged that v0.6.0+ recommendations are speculative because the plugin has no telemetry or user-feedback channel to validate "user demand" claims. Before committing to ANY of the items below, the right move is to add a feedback-collection mechanism (GitHub Discussions, an issue-template for "missing mode", or a usage-survey link in the README).

Conditional candidates (only after evidence of demand):

### Strategic gaps the team flagged that are NOT roadmap items but should be on the maintainer's radar

(All of these came from the architect reviewer in round 2. None require code changes; they require the maintainer to think about scaling.)

1. **Mode count is at or near the sustainable ceiling.** The maintenance cost of 34 modes (10-place invariant × 34 = 340 synchronized artifacts) is already high. Adding mode 35 should be much harder to justify than adding mode 5 was. The bar for "yes, build this" should rise as mode count grows.
2. **No mode deprecation path.** If telemetry eventually shows that some modes (e.g., `cryptanalytic`, `formallogic`) are rarely used, there is no documented process for sunsetting them. Worth designing before the question becomes urgent.
3. **Smoke test scalability.** Already 30-60 min for 34 modes; will not scale past ~50 modes without parallelization (which is in the v0.5.x quality items above) or selective re-runs (only modes whose schemas changed since the last release).
4. **Hybrid mode evolution.** As mode count grows, `hybrid.primaryMode` enum must expand. At some point hybrid may need to become the default entry point (auto-route to specialists from a hybrid root) rather than a sibling mode. Worth a design pass before mode 40.
5. **No telemetry / user feedback channel.** All "consider for v0.6.0+ if user demand" recommendations in this roadmap are speculative because the plugin has no way to measure which modes are actually used or what's missing. A GitHub Discussions board or a periodic README-linked usage survey would let future roadmap decisions be evidence-based instead of speculative.

## Out of scope (deliberate non-goals)

**Mode count beyond ~40:** The current 34-mode taxonomy already covers the major reasoning paradigms. Additional modes should fill clear gaps rather than subdivide existing categories. Reject modes that are primarily academic exercises without clear software engineering applications.

**Formats requiring runtime servers:** Avoid output formats that require persistent server processes or complex runtime environments. The plugin should remain self-contained with minimal external dependencies beyond common command-line tools.

**Domain-specific reasoning modes:** Reject highly specialized modes like medical diagnosis, legal reasoning, or financial modeling that serve narrow domains rather than general analytical thinking. The plugin targets software engineers and technical professionals, not domain specialists.

**Interactive visualization formats:** Avoid formats requiring complex client-side interaction beyond basic HTML dashboards. The plugin focuses on static analysis artifacts rather than interactive exploration tools.

**Proprietary format dependencies:** Reject formats tied to specific commercial tools or requiring proprietary software for rendering. Maintain compatibility with open-source toolchains and standard formats.

## Appendix: How this artifact was generated

This roadmap was generated using a multi-stage RLM methodology: Haiku performed initial extraction passes over 101 reference files to identify 350+ candidate reasoning methods and output formats, then Sonnet synthesized these findings against the canonical 34/9 sets to produce structured recommendations. The analysis prioritized methods with multiple citations, established academic foundations, and clear software engineering applications while filtering out narrow specializations and redundant techniques.
